# Skill: commit_db_write

## Purpose

Execute a previously generated write plan. Requires a valid
`confirmation_token` issued by `plan_db_write` and presented by the human
user. Performs token expiry validation and concurrent-edit conflict detection
before executing the SQL statements transactionally.

## When to Invoke

Only after **all** of the following conditions are met:
1. `plan_db_write` has been called and returned a `plan_id` + `confirmation_token`.
2. The human user has explicitly provided or confirmed the `confirmation_token`.
3. The plan has not expired.

## Do NOT Invoke When

- No write plan exists yet – call `plan_db_write` first.
- The `confirmation_token` was not provided by the user – use `ask_user` to
  request it before invoking this tool.
- The plan was generated with `dry_run: true`.

## Inputs

| Field                | Type  | Required | Description                                          |
|----------------------|-------|----------|------------------------------------------------------|
| `write_plan_id`      | `str` | Yes      | UUID of the write plan generated by `plan_db_write`  |
| `confirmation_token` | `str` | Yes      | User-provided token from the write plan response     |
| `confirmed_by`       | `str` | No       | Identity of the confirming user (audit trail)        |

## Error Cases

| Error                         | Cause                                        | Resolution                          |
|-------------------------------|----------------------------------------------|-------------------------------------|
| `token_expired`               | Token TTL exceeded (default 300 s)           | Re-run `plan_db_write`              |
| `token_invalid`               | Token mismatch or already used               | Re-run `plan_db_write`              |
| `concurrent_conflict`         | Row modified by another process since plan   | Re-run `plan_db_write` + re-confirm |
| `plan_not_found`              | `write_plan_id` does not exist               | Re-run `plan_db_write`              |

## Notes

- Each confirmation token is **single-use**; a committed token cannot be
  replayed.
- All commits are wrapped in a database transaction; on failure the full
  rollback from the write plan is executed automatically.
- Commit events are written to the audit log regardless of outcome.
